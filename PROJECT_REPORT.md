# Project Report: Unified Fraud and Image Authenticity Detection System

## 1. Executive Summary

This project started as an adversarial attack detection system and has evolved into a broader image-authenticity and fraud detection platform. In its current form, it analyzes uploaded images and decides whether they are:

- `CLEAN`
- `AI_GENERATED`
- `ADVERSARIAL_ATTACK`
- `TAMPERED`
- `DUPLICATE_FRAUD`
- `HYBRID_THREAT`

The system combines classical image forensics, feature-space anomaly detection, duplicate checking, and optional persistence for analysis history. The main user-facing product is a Flask web application, with an additional mobile client that talks to the same API.

## 2. Ultimate Goal of the Project

The ultimate goal of this project is to verify whether an image submission is trustworthy.

In practical terms, the project is trying to answer:

1. Is this image a real, legitimate image?
2. Was it generated by AI?
3. Was it adversarially manipulated to fool a model?
4. Was it edited or tampered with?
5. Is it a duplicate or reused submission?

This makes the project useful for fraud-sensitive workflows where image authenticity matters, such as proof-of-delivery, user-submitted evidence, moderation pipelines, or security review systems.

## 3. How the Project Works End to End

### User flow

1. A user uploads one or more images through the web UI or mobile client.
2. The Flask backend receives the image.
3. The backend runs forensic image analysis:
   - FFT spectrum analysis
   - ELA (Error Level Analysis)
   - statistical analysis
   - texture analysis
4. The backend optionally runs the adversarial head:
   - extract an EfficientNet embedding
   - compare it against a clean-image reference database
   - compute LID and Mahalanobis distance
5. The backend checks whether the image is a duplicate:
   - exact SHA-256 hash match
   - near-duplicate cosine similarity in embedding space
6. The unified detector fuses all signals into one final verdict.
7. The result is returned to the frontend.
8. If configured, the analysis summary is saved to history storage.
9. If Redis is configured, repeated analyses of the same image can be served from cache.

### Decision logic

The final verdict is not based on one detector alone. The system intentionally combines multiple signals:

- forensic evidence for AI-generation or tampering
- manifold deviation for adversarial or out-of-distribution behavior
- duplicate evidence for reuse fraud

This is important because any one method on its own can miss edge cases. The project is designed as a layered detection system rather than a single-model classifier.

## 5. What Database Is Used Here

There are three different storage concepts in this repo, and it is useful to separate them clearly.

### A. Primary application database: Supabase / PostgreSQL

The actual persistent application database is Supabase, which is built on PostgreSQL.

Evidence in the code:

- `web/history_store.py` creates a `SupabaseHistoryStore`
- `web/supabase_schema.sql` defines the `analysis_history` table
- the backend talks to Supabase over REST using the project URL and API key

The stored data includes:

- filename
- verdict
- short verdict
- ensemble score
- preview image URL
- FFT spectrum URL
- ELA heatmap URL
- `image_info` as `jsonb`
- `detector_scores` as `jsonb`
- timestamp

### Why PostgreSQL/Supabase makes sense here

Supabase/PostgreSQL is a strong fit because:

1. The stored history is structured and relational.
2. The project benefits from timestamps, indexing, and predictable querying.
3. `jsonb` is useful because detector output is semi-structured and can change over time.
4. Supabase gives a quick hosted backend without needing to manage database infrastructure manually.
5. The app can use simple REST calls instead of adding a heavier ORM layer.

### B. Optional cache: Redis

Redis is also used, but it is not the main database.

In `web/app.py`, Redis is used as an optional cache for image analysis results. The cache key is based on the SHA-256 hash of the image bytes. If the same image is analyzed again, the backend can return the cached result quickly instead of recomputing everything.

Why Redis is used:

1. It reduces repeated computation for the same image.
2. It speeds up the API response for duplicate requests.
3. It keeps caching separate from long-term persistence.

So Redis here is a performance layer, not the source of truth.

### C. Reference embedding database: FAISS + NumPy checkpoint files

The project also contains a reference database for adversarial detection, but this is not a business-data database like PostgreSQL.

It is stored in:

- `checkpoints/reference_embeddings.npy`
- `checkpoints/reference_stats.npz`
- `checkpoints/reference.faiss`

This reference store is used by `LIDDetector` to compare a new image against clean-image embeddings. It supports:

- nearest-neighbor lookup
- LID computation
- Mahalanobis distance computation

Why this exists:

1. Adversarial detection needs a clean reference manifold.
2. FAISS is efficient for nearest-neighbor search in embedding space.
3. NumPy checkpoint files make the reference statistics easy to save and reload.

So if someone asks "what database does this project use?", the best accurate answer is:

- persistent app database: Supabase/PostgreSQL
- cache: Redis
- embedding/reference database for detection: FAISS + NumPy checkpoint files

## 6. Why the Architecture Uses Multiple Detection Methods

The project does not trust one signal alone.

### Forensic detectors

These are useful for spotting synthetic-image artifacts and editing traces:

- FFT can expose abnormal frequency patterns
- ELA can expose unusual compression consistency
- statistics can expose suspicious low-level distributions
- texture can expose overly uniform or synthetic surfaces

### Embedding-space anomaly detection

LID and Mahalanobis distance are useful because adversarial and highly synthetic images often sit far away from the manifold of clean natural images, even when they appear visually plausible.

### Duplicate detection

Fraud is not only about generation or tampering. Reusing the same image can itself be suspicious. That is why the system checks both:

- exact duplicates by hash
- near duplicates by embedding similarity

This layered approach is the main strength of the system.

## 9. Limitations and Practical Notes

### Important limitations

1. The older README files still describe the project mainly as an adversarial detection milestone, while the live code now does more than that.
2. Supabase history is optional, so the app can run without a database, but then history is disabled.
3. The adversarial head depends on the reference checkpoint files being present and loadable.
4. Some pieces in the repo represent older milestone work, while others represent the newer unified system.

### What this means

The most accurate description of the current project is:

"A unified image authenticity and fraud detection system that combines AI-generation detection, adversarial detection, tamper analysis, duplicate detection, and optional history persistence."

## 10. Final Conclusion

This project is not just a classifier and not just an adversarial-attack demo anymore. It is a multi-layer image verification system.

Its ultimate purpose is to determine whether an image submission can be trusted. To do that, it combines:

- forensic image analysis
- embedding-based anomaly detection
- duplicate detection
- optional persistent history and caching

The main persistent database is Supabase/PostgreSQL because it is a good fit for storing structured analysis history with flexible JSON fields. Redis is used for caching repeated analyses, and FAISS plus NumPy checkpoint files act as the clean-reference database for adversarial detection.
